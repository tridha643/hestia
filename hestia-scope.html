hestia▌

A TUI + CLI that gives every git worktree its own isolated stack — containers and host processes (wrangler dev) — so parallel coding agents never fight over ports, databases, or bindings. Hunk for infra. Concurrency capped at 5 for v1 (configurable later).

Verdict

FEASIBLE — VALIDATED SPACE

Yes, this is possible — and every hard sub-problem has been solved in production by at least one existing tool. Docker Compose already supports full per-project isolation via -p / COMPOSE_PROJECT_NAME (separate containers, networks, volumes). Three shipping tools (container-use, docktree, worktree-compose) prove the per-worktree isolation model works and that agents want it. What doesn't exist yet is the TUI — none of them have a hunk-quality interactive surface, and lazydocker isn't worktree-aware. That's hestia's actual product: the isolation engine is mostly plumbing; the differentiation is the cockpit + the agent session protocol + the concurrency governor.

The core mechanic (why it works)

Docker Compose namespaces everything by project name: containers become <project>-<service>-1, each project gets its own bridge network, and volumes are prefixed. So the isolation recipe per worktree is:

# deterministic project name from repo + worktree
project = slug(repo) + "-" + slug(worktree-branch)   # e.g. hestia-salem

# generate an override file — never touch the user's compose file
docker compose -p hestia-salem \
  -f docker-compose.yml \
  -f .hestia/override.yml \   # remapped host ports, injected env, volume names
  up -d

The only genuinely shared, contended resource is host ports — and there are two proven answers, which hestia should combine:

  • HTTP services → named URLs, no ports at all (portless model). vercel-labs/portless (10k★) runs a local HTTPS proxy on :443, gives every app a stable myapp.localhost URL, and — crucially — already detects git worktrees, prepending the branch as a subdomain: salem.myapp.localhost. Backends listen on ephemeral ports (4000–4999) injected via PORT; humans and agents only ever see the stable URL (PORTLESS_URL). Hestia can publish each container port to an ephemeral host port and register it as a route — web.salem.myrepo.localhost — instead of doing port-band math.
  • Raw TCP services (postgres, redis, kafka) → ephemeral ports + env injection. A Host-header proxy can't route non-HTTP traffic, so these still need distinct host ports — but they can be ephemeral (publish port 0, read back what Docker assigned) as long as hestia injects the resulting DATABASE_URL/REDIS_URL into the stack and reports them in hestia status --json. Fixed port bands (docktree's 41000–49999) remain a fallback for tools that need predictable ports.

Everything else (networks, volumes, container names) Docker isolates for free.

The concurrency cap (5 for now, configurable) is not a Docker problem at all — the daemon happily runs hundreds of containers. It's a governance feature: a daemon-held semaphore. hestia up requests a slot; if 5 stacks are live, the request queues (or fails fast with --no-wait), and agents see their queue position via --json. The real ceiling is machine resources (Docker Desktop VM RAM on macOS, plus one workerd process per worktree), which is exactly why an explicit cap + visible queue is the right design instead of letting a fleet of agents thundering-herd the daemon. Starting at 5 also keeps duplicated Postgres instances and workerd processes comfortably inside a laptop's budget; raising it later is a config change, not an architecture change.

Cloudflare Workers & tunnels (the modem constraint)

The modem repo means stacks aren't just containers — wrangler dev runs workerd as a host process, not in Docker (Cloudflare's own issue tracker shows containerizing it breaks service bindings). So hestia's stack unit must be compose services + native processes, process-compose style. The wrangler-specific isolation surface, per worktree:

Traffic flow when the repo isn't wired

Hestia injects at process and env boundaries — ports via flags/overrides, URLs via env — so most traffic routing lives outside the repo and works on an untouched checkout. The public-ingress trick: hestiad owns the tri tunnel (one tunnel, one connector, no replicas) with a wildcard hostname pointed at hestia's proxy, which routes by Host header to the right worktree:

internet ── tri tunnel (hestiad-owned, ingress: *.dev.modem.dev)

The callback router (accounting for webhooks & OAuth)

Since hestiad terminates the canonical public URL, callback handling is a routing problem inside hestiad, not a repo problem. Four mechanisms, composable, in escalating order of precision:

What remains genuinely un-accountable from outside: two stacks whose source code insists on binding the same 127.0.0.1:3000. That's the one case where the repo must move to env-driven config — and hestia doctor exists to find those spots: it scans source for hardcoded ports/hostnames, checks compose and wrangler configs, and prints exactly what works untouched vs. what needs the env-var treatment, so wiring the repo is a 20-minute checklist, not archaeology.

Architecture (steal hunk's shape)

1 · isolation engine

Pure library, two backends behind one stack abstraction: docker compose (override generation, volume names, env injection) and supervised host processes (wrangler dev with injected --port/--inspector-port/WRANGLER_REGISTRY_PATH). Plus worktree discovery, deterministic naming, stale-stack GC.

2 · hestiad (session broker)

Loopback daemon, same as hunk's session-broker. Owns the concurrency semaphore (default 5), the queue, stack registry, health/event stream. TUI and CLI are both just clients.

3 · surfaces

TUI (OpenTUI, like hunk) for humans. CLI with --json everywhere for agents, plus hestia skill path shipping a SKILL.md that teaches agents the protocol.

Agent flow mirrors hunk exactly: agent runs hestia up --json in its worktree → daemon allocates a slot + URLs/ports → agent gets back service URLs and env → runs its e2e tests → hestia down releases the slot. Human watches all 5 stacks in the TUI and can jump into any stack's logs or exec a shell.

Hard parts (honest list)

SOLVED · project isolation

Compose -p handles containers/networks/volumes natively. Generated override files keep the user's compose file untouched. docktree/wtc both ship this.

SOLVED · concurrency cap

A semaphore in hestiad, hard-set to 5 for v1 (config-file override lands later). Queue with position feedback in --json. Trivial to build, high perceived value for agent fleets.

MEDIUM · ports & URLs

HTTP: adopt the portless model (or integrate portless itself) — stable svc.worktree.repo.localhost URLs via a local proxy, ephemeral backend ports. TCP (postgres/redis): ephemeral host ports + injected connection env; fixed bands as an opt-in fallback. Apps that build URLs inside runtime shells (Vault/Doppler) can't be rewritten safely — document the escape hatch. One portless gotcha to inherit: dev-server proxies must rewrite the Host header or requests loop.

MEDIUM · docker network address-pool exhaustion

Each compose project creates a bridge network; default Docker address pools run out around ~30 networks. At 5 stacks it's comfortable, but stale networks accumulate — need hestia gc + pool-exhaustion detection (docktree literally ships --prune-networks for this exact failure).

MEDIUM · resource pressure on macOS

5 stacks × 3–5 services ≈ 15–25 containers plus a workerd + vite process per worktree. Duplicated Postgres instances are the heavy part. Mitigations: per-service memory limits in the override, and later a "shared services" tier (one Postgres, DB-per-worktree) like docktree's tenancy: per_database. The cap of 5 is the primary guardrail here.

MEDIUM · wrangler dev registry cross-wiring

Service bindings resolve by worker name through a shared filesystem registry — parallel worktrees silently RPC into each other's workers. Fix is cheap (WRANGLER_REGISTRY_PATH per stack) but must be automatic and verified, since the failure mode is silent wrong-answers, not a crash.

MEDIUM · named-tunnel replica cross-wiring

Same failure family: reusing one cloudflared tunnel ID across worktrees makes them HA replicas and Cloudflare round-robins the hostname between stacks. Hestia must mint a tunnel per worktree and clean up its DNS route on down. Quick tunnels dodge this entirely and are the agent default; any tunnel is opt-in + badged since it's public exposure of an agent-run stack.

HARDEST · lifecycle hygiene

Worktree deleted while its stack runs; agent crashes without down; volumes orphaned. The daemon must reconcile: watch worktree list, TTL-reap idle stacks, and make gc a first-class TUI action. This is where naive versions rot — and where a persistent daemon (vs. wrapper scripts) earns its keep.

MVP cut

1
Engine + CLI (agents first, modem-shaped).

hestia up/down/status --json, project naming, compose override generation, and the process backend for wrangler/vite (injected ports, per-stack WRANGLER_REGISTRY_PATH) — modem needs both backends on day one. Semaphore (cap 5) in a minimal daemon, hestia skill path, and hestia doctor (report-only) to surface what the repo needs wired. This alone is agent-usable — ship it before any UI.

2
TUI cockpit.

OpenTUI app: worktree sidebar, service pane, log tail, queue indicator, u/d/s/l keybindings. Reuse hunk's session-broker packages (they're published as runtime-neutral @hunk/session-broker*).

3
Named URLs + ingress (portless layer).

Integrate or embed a portless-style proxy: svc.worktree.repo.localhost locally, HTTPS via local CA, injected URL env — and the hestiad-owned tunnel with wildcard Host routing for public traffic. Callback router v0 rides along: event journal + hestia events replay + broadcast mode.

4
Hygiene + scale.

GC/reaper, network-pool pruning, resource limits, health/event stream, per-stack shell attach.

5
Later.

Callback router v1 (OAuth affinity cookie, payload-match rules, Stripe/GitHub auto-registration), shared-services tier (one Postgres, DB per worktree), MCP server alongside the CLI, "promote winner" flow (wtc-style), LAN/Tailscale sharing, non-Docker backends (podman, container-use/Dagger).
